#89 — HIGH — /school/layout.tsx auth-race retry (final layout!)
============================================================
Same race shape as the other three layouts:
  Round 8 #53  — /teacher/layout.tsx
  Round 16 #72 — /student/layout.tsx
  Round 17 #77 — /admin/layout.tsx
  Round 20 #89 — /school/layout.tsx  ← THIS

Single /api/auth/me call with no retry was bouncing real
super_teachers to /login during the 300-800ms auth-server
propagation window right after sign-in. Now uses the same 3-attempt
× 300ms-backoff pattern.

With this commit, **every layout in the platform handles the
auth-race consistently**. No layout-level first-sign-in bounces
should remain across teacher / student / admin / super_teacher.

#90 LOW /school/billing client-side error handling
  When session.access_token is missing, the page shows "Sign in
  required." instead of redirecting to /login. Layout already
  redirects, so client-side is defensive only. Cosmetic.

#91 MEDIUM /school/students N+1-style query pattern
  Builds the roster via 5 sequential supabaseBrowser() queries:
    (1) /api/auth/me        (resolve school_id)
    (2) profiles            (students in school)
    (3) class_members       (their memberships)
    (4) classes             (active classes in school)
    (5) attempts            (their attempts on school quizzes)
  Each is ~50-200ms RTT — page total load can hit ~500ms-1s.
  Fix: server-side aggregator endpoint /api/school/students that
  returns rows + memberships + attempts in one call. Out of scope
  for this commit; the page works correctly today, just slow.

#92 LOW /school/billing UI doesn't surface the grace_period_days
  field even though the API returns it. Operators add grace_period
  for late payers; school doesn't see "you're 7 days past due but
  in grace until DD-MM". Add when finance team requests visibility.

#93 LOW /school/reports.tsx is 1279 lines — needs decomposition into
  smaller components before the next feature pass. Architectural,
  not a bug.
