#79 — MEDIUM — /admin/plans/queue/[id] stale-edit detection
============================================================
Symptom: two admins open the SAME open proposal in different tabs.
Admin A clicks "Edit and approve", iterates the payload, saves.
Admin B (still on stale state from initial load) clicks Approve →
the request includes Admin B's stale snapshot as the "edits" body,
which silently overwrites Admin A's fresh save with the old version.
Same shape applies to reject / withdraw / saveDraft.

Fix: new ensureFresh() helper called at the top of every
approve/reject/withdraw/saveDraft action. It re-fetches the proposal
via GET /api/admin/plan-proposals/[id], compares the live `proposed`
payload (deep-equal via JSON.stringify) AND the live `status`
against what's in component state. If they differ, prompt the user:

  "Another admin set this proposal to 'approved'."          (status drift)
  "Another admin saved an edit to this proposal."          (payload drift)
   [OK to reload]  [Cancel to abort]

Fail-open on read errors — a transient network blip must NEVER
block a real action.

#81 LOW /api/admin/onboard-school school-insert + profile-update race
  Status: documented; not fixed.
  The two writes (schools INSERT + profiles UPDATE) happen
  sequentially via Supabase JS. The compensating rollback already
  handles the case where the profile-update fails (delete the
  school + delete the auth user). A true transaction would tighten
  this further, but Supabase JS doesn't expose multi-statement
  transactions; the proper fix is a Postgres RPC that does both in
  one call. Not worth the complexity until we have evidence the
  narrow race window is hurting onboarding.

#82 LOW /admin/plans/queue/[id] bearer() helper redundant getSession()
  Status: documented; not fixed.
  Each action (approve / reject / withdraw / saveDraft) calls
  bearer() which calls supabaseBrowser().auth.getSession(). Supabase
  JS caches the session internally, so this is a JS-allocation
  cost (no network). Not measurable in production.
