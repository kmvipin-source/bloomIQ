1. L-1 — signup "Database error saving new user" (CRITICAL)
============================================================

Both /signup?role=teacher AND /signup?role=student were failing with
"Database error saving new user" on the dev environment. Root cause:
the public.handle_new_user() trigger does 5 things in sequence:

  1. Validate role
  2. INSERT into profiles
  3. INSERT into subscriptions (independent students only)
  4. UPDATE profiles.school_id from class_teacher_invites
  5. INSERT class_teachers + DELETE class_teacher_invites

Steps 1-3 are mandatory. Steps 4-5 are best-effort auto-claim of
pending teacher invites. The old trigger aborted on any failure
ANYWHERE in steps 4-5 — including a missing/stale class_teacher_invites
table on a partial-migration dev DB.

Fix: supabase/migrations/99_handle_new_user_defensive.sql
  - Steps 1-3 unchanged (same happy path as migration 97)
  - Step 3 wrapped in BEGIN…EXCEPTION (subscriptions schema drift)
  - Pre-checks information_schema for class_teacher_invites before touching it
  - Wraps steps 4-5 in BEGIN…EXCEPTION WHEN OTHERS so any failure logs
    via RAISE NOTICE but does NOT abort the trigger
  - CREATE OR REPLACE, idempotent

After `supabase db push`, signup will succeed even if the invites
table is in an unexpected state.

2. L-2 — /api/student/quick-test had NO rate limit and NO daily cap
============================================================

The student "Take a test" endpoint fans out to:
  - up to 6 LLM generation calls (one per Bloom level)
  - a verifier pass per question (~5 reviewer calls each)
  - optional refinement on dispute
  - an auto-retry on per-level shortfall

A scripted abuser could burn the day's Groq+Gemini budget in one loop.

Fix: app/api/student/quick-test/route.ts
  - import checkRateLimit, checkDailyCap from "@/lib/rateLimit"
  - after auth+role-check: bucket(capacity=5, refill=10/hr) + daily cap=30
  - 429 deny responses mirror the shape /api/generate already uses

Limits chosen to match /api/generate (same cost shape, same pipeline).

============================================================
3. L-3 — Build & Assign composer never wrote to quiz_assignments
============================================================

The /teacher/quizzes/new composer let teachers pick a class for the
class-fit warning, but never actually assigned the new test to that
class. The teacher had to navigate to a second screen and re-pick the
same class to make it visible to students. Many teachers were missing
this step entirely.

Fix: app/teacher/quizzes/new/page.tsx
  - new assignDueAt state on ComposerInner
  - new optional datetime-local input on the ClassFitCard, only shown
    when a class is picked (with a helper line explaining what happens
    on save)
  - in create(): after quiz_questions insert succeeds, if targetClassId
    is set, insert a quiz_assignments row (whole-class, due_at=null
    when blank — migration 01 allows nullable due_at)
  - failure is non-fatal — quiz still exists; failure surfaces inline
    so the teacher can retry assigning from the detail page

4. U-2 — /api/signup-and-pay signin-failure rollback (carries forward)
============================================================

Static workflow trace: if signInWithPassword fails between the
profile-upsert and the Razorpay order-create, the auth user + profile
rows both exist but the user can't sign in. No compensating rollback.
The email-exists check then blocks re-signup. User is locked out,
needs manual support to recover.

Fix: app/api/signup-and-pay/route.ts
  - wrap signIn in try/catch
  - on failure call admin.auth.admin.deleteUser
  - matches the rollback pattern already used by the Razorpay-order
    failure path further down the same handler

5. AI Quality Benchmark Harness (carries forward)
============================================================

Three-step pipeline to empirically validate AI question quality against
51 production-shaped prompts. Located under scripts/ai-quality-benchmark/

  prompts.json   — 51 prompts: K-12, JEE/NEET/CAT/UPSC/GMAT/GRE/IELTS/
                   SAT/BITSAT, disambiguation traps (LCM, ROI, AC, PI, F),
                   niche domains (Cobol, ISO 8583, BGP, DB2), mock-paper
                   triggers, cross-validation traps, edge cases.
  run.mjs        — sends each prompt through POST /api/generate
                   (full pipeline). 2s pause between prompts; 30s pause
                   + retry on 429.
  grade.mjs      — emits two CSVs for educator grading
  summarize.mjs  — reads filled CSVs, computes per-dimension and
                   per-category acceptance, compares against 85% pass
                   bar, exits 0=pass, 2=fail.
  README.md      — full operator instructions and re-run cadence.
